# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Calmecac is a from-scratch LLM agent harness — a learning project. The backbone is a local
Ollama server (native `/api/chat`, not the OpenAI-compatible endpoint) at `http://localhost:11434`,
default model `qwen3:14b`, configured in `config.yaml`.

## Working constraints (read before editing)

This project is built in small, deliberate phases and the code is meant to be fully understood
line by line, not generated by a framework:

- **No agent frameworks** (no LangChain/LangGraph/CrewAI, etc). Allowed dependencies: `httpx`,
  `pyyaml`, and stdlib. `pydantic` is allowed but not yet used — don't add it speculatively.
  SQLite is planned for logs eventually but not yet — logs are JSONL files for now.
- **Boring, explicit code over clever/generic code.** No class hierarchies, no plugin systems,
  no callback/generator abstractions where a direct function call and a `print()` will do.
- **Don't build ahead of the current phase.** Don't add abstractions, config options, or
  placeholder scaffolding for features that don't exist yet (e.g. skill loading, MCP, retries,
  streaming). If something seems missing, ask before adding it rather than assuming.
- **Never silently repair malformed model output.** If the model produces a bad tool call
  (unknown tool, wrong arguments, non-JSON arguments), the failure is reported back to the model
  as an error message so *it* can retry — the harness must not guess what was meant.

## Commands

```bash
# Smoke test: sends "say hello" through harness/ollama_client.py
python -m harness.smoke

# Run the agent loop on a task
python -m harness.run "your task here"
```

There is no test suite, linter, or build step configured yet — verification so far has been
direct execution against a live Ollama instance (see `logs/` for transcripts of past runs) and
one-off `python -c` checks of individual functions (e.g. `harness.tools` path-escape guards,
`harness.loop._call_tool` error handling).

## Architecture

**Data flow:** `harness/run.py` (CLI) loads `config.yaml` → calls `harness/loop.run_task(task, config)`
→ which loads the system prompt from `agents/kukulkan/prompt.md`, then drives the ReAct loop via
`harness/ollama_client.chat(messages, tools)`, executing any tool calls through
`harness/tools.REGISTRY` and appending results back into the message list until the model returns
plain text (or `max_steps` from `config.yaml` is exhausted).

- **`harness/ollama_client.py`** — the only code that talks to Ollama. One function,
  `chat(messages, tools=None)`, posts to `/api/chat` with `stream: false` and returns the raw
  parsed JSON. It loads `config.yaml` once at import time for `base_url`/`model`/`temperature`/`num_ctx`.
  Note: Ollama's native tool-calling parses `<tool_call>` tags out of the model's output into a
  structured `message.tool_calls` field — this depends on the *model* reliably following its
  chat template. `qwen2.5-coder:14b` was observed to fail this consistently (emits raw JSON as
  plain `content` instead), which is why `qwen3:14b` is the configured default.

- **`harness/tools.py`** — tool implementations as plain functions (`read_file`, `write_file`,
  `list_dir`), all routed through `_resolve()` which confines every path to `workspace/`
  (rejects absolute paths and `..` traversal — note that `Path("a") / "/etc/passwd"` silently
  discards the base in pathlib, so absolute paths must be checked explicitly, not just resolved).
  `REGISTRY` maps `{tool_name: (json_schema, callable)}` — the schema is what gets sent to
  Ollama's `tools` parameter, the callable is invoked directly via `func(**arguments)`. Adding a
  tool means adding one function plus one `REGISTRY` entry, nothing else.

- **`harness/loop.py`** — the ReAct loop (`run_task`). One iteration of the `for step in
  range(1, max_steps + 1)` loop is one model call, which may contain zero or more tool calls.
  Every model call and every tool call/result is appended as a JSON line to
  `logs/run_<UTC-timestamp>.jsonl` — this is the primary debugging surface (read manually, not
  through tooling). `_call_tool` returns `(result, error)` with exactly one populated; malformed
  arguments become an error string appended as the tool result, not a fixed-up call. Hitting
  `max_steps` returns an "aborted" string rather than raising, matching the `-> str` return type.

- **`agents/<name>/prompt.md`** — one directory per agent persona, containing just the system
  prompt. Currently only `agents/kukulkan/`. Not templated or parsed — read as raw text.

- **`skills/`** — empty. Intended format (see `skills/README.md`): one markdown file per skill,
  `skills/<skill-name>.md`. Not loaded by any code yet.

- **`evals/`** — empty placeholder; actual eval harness lives in a separate project (link TODO
  in `evals/README.md`).

- **`workspace/`** and **`logs/`** — gitignored, created on demand at runtime (`workspace/` by
  `harness/tools.py`, `logs/` by `harness/loop.py`). This is where tool file I/O and run
  transcripts actually land.
